GitHubを利用した分散開発フロー

開発者種別

conkan開発協力者は、その権限により下記の3種類に分類される。
  • リポジトリ管理者
    GitHub conkanリポジトリのすべての操作権限をもつ。
    (実際には、conkan Organizationのowner権限を持つ者)
    新たなリポジトリの作成や、チーム開発者の登録を受け持つ。
  • チーム開発者
    GitHub conkanリポジトリへの書き込み権をもつ。
    (実際には、conkan Organizationのcollaboratorsチームに属している者。リポジトリ管理者を含む)
    GitHub conkanリポジトリでのブランチ作成、developブランチへのマージを受け持つ。
  • 外部開発者
    GitHubのアカウントを持ち、conkanの開発に協力する。
    (conkanの開発に参加するためには、GitHubのアカウントがあるだけで良い)
    GitHubのIssueにコメントする(コードレビュー)
    GitHub conkanリポジトリを自分のリポジトリにforkし、PullRequestを発行する。

外部開発者による開発フロー

リポジトリを使った分散開発にあたり、各開発者は原則として下記の手順を取る。
  1. 対応するIssueに、自分が担当する旨明記する。
  2. 自分の開発環境で、developブランチから実装用ブランチを作成し、実装を実施する。
    チーム開発者であればgit flowモデルに従う。
    そうでない場合は、対象リポジトリを自分のGitHubリポジトリにforkする。forkしたリポジトリの作業フローは任意。
  3. 実装/評価が完了したら、conkan-GitHubリポジトリのdevelopブランチにマージするよう、PullRequestを発行する。
    もちろん、実装中や評価前に、議論を目的にPullRequestを発行するのが望ましい。
    その場合、PullRequest名で作業中であることがわかるようにすること。(習慣的に、先頭に[WIP]を付ける)
  4. リポジトリ管理者は、PullRequestの内容をチェックし、問題がなければdevelopブランチにマージする。
    developブランチは、常に開発版としてデプロイし運用可能であることを保証する。

リポジトリ管理者は、適切なタイミング(年1度、など)でreleaseブランチを作成し、正式版としてリリースする。
リリース版はconkan動作環境構築方法で利用できる形式で提供する。
ただし、conkan.github.ioは例外であり、developブランチの内容をmasterブランチにマージすることで、デプロイ/リリースを同時に実施する。
